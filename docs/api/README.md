# API Docs

本目录维护项目的接口契约——后端的生产方契约，以及前端的消费方契约。

## 权威来源

- 后端事实源：接口定义文件（如 IDL / proto / OpenAPI）+ 代码实现。
- 人类可读契约：本目录下的接口契约文档（按项目填写文件名，如 `api-front.md`）。
- 如有前端镜像文档，修改后端契约时需要同步前端镜像。

## 通用约定

> 以下为滴滴内网项目的默认约定示例，按项目实际接口风格调整。

- 业务接口统一使用 `POST + JSON body`。
- 资源 ID 放 request body，不使用 path 参数。
- 响应统一包裹为 `errno`、`errmsg`、`data`。
- `errno === 0` 才表示业务成功。
- 时间字段遵循接口契约中的 DateTime / Date / Time 规则。
- 错误码以接口契约中的通用错误码表为准。

## 前端消费契约（前端视角）

前端不只镜像后端契约，还要记录「数据怎么落到界面」：

- **字段 ↔ 展示映射**：每个字段渲染成什么（文案 / 标签 / 状态色 / 格式化规则）。
- **错误码 ↔ 错误态**：哪个 `errno` 对应页面的哪种错误态（toast / 整页错误 / 字段级提示）。
- **各态数据条件**：空态 / 加载态 / 异常态分别由什么数据条件触发。
- **mock 约定**：联调前的 mock 数据来源与字段示例。

建议每个对接模块一个 `<模块>.md`（或前端镜像 `<模块>-front.md`）；后端契约变更时，这里的展示映射要同步核对。

## 维护规则

- 新增接口时先更新接口定义和实现，再更新本目录文档。
- 字段命名、错误码、分页、鉴权规则必须写在契约中。
- 不把业务背景长文放进 API 文档；背景和设计放到 [`../overview/`](../overview/) 或 [`../design/`](../design/)。
